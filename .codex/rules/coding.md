# Coding Rules

## 职责

- 本文件只定义代码修改前置条件、修改边界、编码风格和禁止事项。
- 本文件不定义 subagent Flow、队列迁移、验证命令、交付记录格式。

## 计划

- 修改代码、排障、多步骤任务必须先给计划。
- 计划格式必须为：`1. [操作] → verify: [验证方式]`。
- 若需求存在歧义且影响实现路径，必须先提问。

## 开发前

- 修改代码前必须读取相关代码，至少包含被修改函数和调用方。
- 明确影响范围：文件、函数、接口、用户可见行为。
- 确认当前任务已有可验证成功标准。
- 修改超过 100 行时，先重新评估是否能拆小。

## 修改边界

- 只修改当前任务直接相关文件。
- 禁止修改无关格式、注释和配置。
- 禁止添加未请求的配置项。
- 禁止新增通用抽象，除非明确复用不少于 2 处。
- 实现超过 100 行时，必须重新评估是否可简化。

## 编码风格

- 使用函数优先，除非已有代码要求 class。
- 使用提前返回，减少嵌套层级；嵌套层级不超过 2 层。
- 变量名必须表达完整语义，禁止 `a`、`b`、`tmp` 这类命名。
- 注释只解释为什么，不描述代码表面行为。
- 删除本次修改引入的未使用变量、函数、导入。

## 禁止

- 主 agent 禁止实现或修改任何业务代码（`client/**`、`server/**`），必须通过 subagent（frontend / backend / integrator）执行；主 agent 仅负责探索、调度、审查和队列维护。
- 未经允许安装或升级依赖。
- 未经允许修改 `.env`、`.gitignore`、密钥、证书、生产配置。
- 未经允许修改数据库结构、接口定义、部署流程。
- 未经允许删除文件。
- 未经允许执行大规模重构，范围为 3 个文件及以上。
